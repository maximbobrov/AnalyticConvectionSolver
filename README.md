# Analytic Convection Solver

A single-file C++/OpenGL (GLUT) prototype simulating axisymmetric (r, z) buoyancy-driven convection. A localized buoyancy source at the bottom boundary drives a vorticity/velocity field through a relaxation solver. The project's name comes from its defining trait: the vorticity boundary condition isn't obtained by resolving the boundary layer numerically, it's taken from a closed-form (analytic) plume-scaling law.

## Physical model

The domain uses cylindrical coordinates (r, z) on a `GRID_NR=100 x GRID_NZ=100` grid, with cell size `CELL_SIZE_R=CELL_SIZE_Z=0.1` m (10x10 m domain). `r=0` is the symmetry axis (left edge of the window).

State fields (`convection.cpp`):

- `vorticity` (Om) — azimuthal vorticity, units `1/s`
- `velocityZ` (Uz), `velocityR` (Ur) — vertical/radial velocity, units `m/s`
- `buoyancy` (B) — buoyancy perturbation, units `m/s^2`

Parameters set in `initSimulation()`: `buoyancyFrequency` (N) = 0.01, `buoyancyDiffusivity` (k) = 1e-2, `viscosity` (nu) = 1e-2.

### Boundary and initial conditions

At the bottom boundary (`z=0`) a fixed buoyancy bump is imposed, peaked at the domain center and decaying radially:

```
B(r, 0) = BUOYANCY_BOUNDARY_PEAK / (|i - GRID_NR/2| + 1)
```

The matching vorticity boundary value follows the analytic scaling law that gives the project its name (a classic result for the vorticity generated by a buoyant boundary-layer plume), instead of being computed from a resolved boundary layer:

```
Om(r, 0) = VORTICITY_SCALING_COEFF * viscosity^(-1/3) * B(r, 0)^(2/3)
```

The top boundary uses a zero-gradient (Neumann) condition. Both are enforced structurally: row `j=0` is set once in `initSimulation()` and never touched again by the relaxation sweep (`relaxImplicitZ` only updates `j = 1 .. GRID_NZ-2`), and row `j=GRID_NZ-1` is copied from `j=GRID_NZ-2` every iteration.

> **Bug fix note:** the original code set `Om[i][0] = bcOm[j]` (using the loop index `j`, which is always `0` in that branch) instead of `bcOm[i]`, so every radial column got the same bottom vorticity value (`bcOm[0]`) rather than its own value from the scaling law. This has been corrected in `initSimulation()` — the boundary vorticity now correctly varies with radius, matching the buoyancy bump's shape.

### Governing equations and numerical scheme

Each call to `advanceTimestep()` sweeps radial rows `i = 0 .. GRID_NR-1` (from the axis outward) and, for each row, solves three coupled equations implicitly in `z` (damped Gauss-Seidel relaxation, `relaxImplicitZ`) while coupling to the neighboring row `i-1` explicitly (already updated earlier in the same sweep):

1. **Vorticity transport** — advection `Ur*Om/r` plus a baroclinic (buoyancy-gradient) torque term, divided by viscosity.
2. **Vorticity -> velocity Poisson relation** for `Uz` — the cylindrical Laplacian of `Uz` sourced by `Om`.
3. **Buoyancy transport** — linearized Boussinesq equation `dB/dt ~ k*Laplacian(B) - N^2*Uz`: diffuses and is forced by vertical advection of the background stratification.

After all rows are updated, `computeRadialVelocityFromContinuity()` recovers `Ur` from `Uz` by integrating the cylindrical continuity equation `(1/r) d(r*Ur)/dr + dUz/dz = 0`, keeping the flow approximately divergence-free.

Time-stepping is explicit/pseudo-transient: while running, each rendered frame performs `TIME_STEPS_PER_FRAME=50` relaxation sub-steps with a small rate `RELAXATION_RATE=0.00005`, so the field evolves slowly and needs many frames to visibly change.

## Visualization

- **Field mesh** (`drawField`) — a triangle-strip colored by the active field's value (scaled by `colorScale`, keys `,`/`.`). For `B`, `Uz`, `Ur`: positive = red, negative = cyan. For `Om`: positive = green, negative = magenta.
- **Axes** (`drawAxes`) — r/z axes with 1-meter tick marks and labels ("r, m" / "z, m").
- **Legend overlay** (`drawLegend`) — fixed to the screen (not affected by pan/zoom): active field name, its units, current min/max value range, and a color-key reminder. Updates automatically when switching fields.
- **Velocity vectors** (`drawVelocityVectors`, toggle key `v`) — small points on a `~VECTOR_GRID_SAMPLES x VECTOR_GRID_SAMPLES` (25x25) grid, each with a thin white line showing `(Ur, Uz)`. Vector length is proportional to velocity amplitude: it's auto-scaled every frame so the fastest vector is `MAX_VECTOR_LENGTH` long (1/30 of the domain width, raw velocities are orders of magnitude smaller than the 10x10 m domain and would otherwise be invisible), then further multiplied by the user-adjustable `vectorLengthScale`.

## Build & run

The original `.pro` file targeted an old 32-bit GLUT (`my_lib/glut32.lib`), which doesn't link with modern 64-bit toolchains. The project now uses [freeglut](https://freeglut.sourceforge.net/) via [vcpkg](https://github.com/microsoft/vcpkg), and the source file was renamed `main.cpp` -> `convection.cpp` to describe what it actually does; the now-unused `my_include`/`my_lib` directories were removed.

```
vcpkg install freeglut:x64-windows

cl /EHsc /O2 /I"%VCPKG_ROOT%\installed\x64-windows\include" convection.cpp /Fe:AnalyticConvectionSolver.exe ^
   /link /LIBPATH:"%VCPKG_ROOT%\installed\x64-windows\lib" freeglut.lib opengl32.lib glu32.lib
```

Copy `freeglut.dll` from `%VCPKG_ROOT%\installed\x64-windows\bin` next to the produced `.exe`.
